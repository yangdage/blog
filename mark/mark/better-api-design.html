为用户节省流量之API接口设计

Notes
2015-02-23
mark

前言

移动互联网时代，在没有WIFI的情况下，用户对流量异常的敏感。往往在打开一个APP的时候，APP后台会请求服务器获取最新的数据，如果返回的数据非常多，就会很费流量。在技术层面可以利用一些技巧来设计请求的API，帮助用户节省不必要的流量消耗，也提高APP自身的可用性。这里以我们常用的QQ获取最新的好友列表为例，当然QQ具体肯定不是这么实现的，QQ客户端跟服务器的交互一般使用TCP/UDP协议建立socket长连接，这里只是记录几种使用TCP/HTTP协议短连接来实现类似功能的方案。

获取好友列表最先想到的实现方法可能就是每次开QQ时都会去拉取一次所有的好友列表数据，数据中包括用户昵称，备注，头像，个性签名，在线状态，等等。然后客户端在一直打开的情况下，还会定时轮询去拉取最新的好友列表，同步到客户端，这种可能也是最耗流量的实现方式了。下面提供两种优化的方式。

两次请求，客户端比对时间戳

还是以上面客户端获取最新好友列表为例，首先，假设客户端已经获取过一次好友列表数据，数据中包含每个用户信息的最后更新的时间戳，然后我们拉取所有好友的最后时间戳数据，跟客户端已有好友时间戳数据比对，找出不同的用户id，也许100个好友中只有10个不同，这样我们接下来只要更新下这10个用户的数据就好了，这样可以节省将近了90%的流量消耗。

这种方式有个缺点就是每次跟新数据都要请求两次服务器，一次获取用户时间戳，一次获取有差异的用户列表数据。

一次请求，服务器比对时间戳

客户端将本地好友ID和时间戳数据全部提交到服务器，服务器负责比较用户好友列表时间戳，返回有更新的用户列表。这种方式会增加服务端的压力。
