LSM-Tree

Notes
2016-04-28
mark

LevelDB是一个开源的键值存储数据库，基于Google的BigTable思想，基于LSM-Tree算法实现，也受到工业界和学术界也来越多的关注和支持。

下图展示的是Level-DB的基础架构，包含2个内存中的MemTable数据结构部分，和一系列磁盘中的SSTables（ Sorted String
Table file）数据结构部分，以及一些其他附属文件，如Manifest文件，用于存储SSTTables的元素据。

每当用户插入一条键值对数据到LevelDB中的时候，首先会在日志文件中记录一条相关日志。然后才会将数据插入到内存中一个叫MemTable的有序的数据结构中。
MemTable维护着最近的数据更新。当数据大小增长到超过其容量的时候，MemTable将被转化为只读的数据结构Immutable MemTable，同时会生成一个新
的MemTable用于接收新的数据。这时，后台会有另外一个线程会将Immutable MemTable中的数据写入磁盘，并重新生成一个排好序的SSTable文件。

SSTable存储的数据按照数据项的键的排序。磁盘中的SSTable文件被分成几个等级，最年轻的等级Level0，由内存中的Immutable MemTable数据写入磁盘时产生。
只有Level0中的数据允许包含重复的键。

每一层级中都有对SSTable文件大小总量的限制，并且限制的大小随着层级的增加呈现指数级的增长。
比如Level1中可存储文件大小总量不超过10M，Level2中可存储的文件大小总量不可超过100M。

为了保证存储的数据结构最优化，后台的压缩线程将实时监控SSTable文件的大小变化，当该层级中的文件总大小超过限制值时，压缩线程将在该层级中选取一个SSTable文件，
并选取下一层级中的所有SSTable文件进行合并处理，重新生成一系列有序的SSTable文件。默认每个SSTable文件的大小为2M，每当新生成的SSTable超过2M时，将重新创建新的SSTable文件。
所有参与压缩的旧的文件将会全部被删除。

Level0到Level1的压缩处理比较特殊，这一层的处理由SSTable文件的数量限制触发，默认时超过4个文件时就会进行压缩操作。

通过压缩操作，可以确保越新的数据会保存在越小的层级中，越老的数据，将会被压到层级的越底层。

读取数据的操作比较复杂。当LevelDB收到读取请求时，首先会去MemTable中查找数据，然后再去Immutable MemTable中查找，最后再去SSTable的分层文件中查找，查找顺序从层级0开始。
