CSRF和XSS防御与攻击

CSRF概述

CSRF(Cross-Site Request Forgery)，翻译过来叫做跨站请求伪造。是一种挟制终端用户在当前已登录的Web应用程序上执行非本意的操作的攻击方法。
攻击者只要借助少许的社会工程诡计，例如通过电子邮件或者是聊天软件发送的链接，攻击者就能迫使
一个Web应用程序的用户去执行攻击者精心构造的操作。

实例解释

假设某用户登录网络银行去查看其存款余额，他没有退出网络银行系统就去了自己喜欢的论坛去逛逛，如果攻击者在论坛事先已经构造了一个恶意转账的链接并诱使该用户点击了该链接，那么该用户在网络银行帐户中的资金就有可能被转移到攻击者指定的帐户中。

危害

当CSRF针对普通用户发动攻击时，将对终端用户的数据和操作指令构成严重的威胁，危及个人隐私及财产；当受攻击的终端用户具有管理员帐户的时候，CSRF攻击可能会危及整个Web应用程序。

CSRF成因

  • Web浏览器对于Cookie和HTTP身份验证信息之类的会话信息的处理方式

浏览器会自动地发送标识用户对话的信息，而无需用户干预。换句话说，
当浏览器发送这些身份信息的时候，用户根本感觉不到。

  • 攻击者对Web应用程序URL的了解

分析请求URL或者通过访问该应用程序并查看嵌入HTML/JavaScript中的URL
以及表单来了解应用程序有关的URL、参数和容许值。

  • WEB应用会话管理对浏览器的透明性

浏览器可以访问会话管理信息，如果Web应用程序完全依赖于这类信息来识别
一个用户会话，这就为跨站请求伪造创造了条件。

  • 各种能够引发资源请求HTML标签

CSRF触发条件

  • 1.登陆信任网站，本地存有用户cookie
  • 2.没有登出信任网站，访问危险网站。

被攻击用户浏览器在用户不知明的情况下被迫向被攻击站点发起伪造的请求。
请求过程会带上被攻击用户的身份认证标识(cookie)。

用户防御意识

  • 使用完Web应用程序之后记得退出登录
  • 不要让浏览器保存敏感网站用户名/密码
  • 不要随便点击陌生人发的未知链接
  • 不要使用同一浏览器同时访问敏感的应用程序和随意冲浪
  • 小心查看支持HTM格式的邮件

CSRF防御

  • 1.过滤所有用户发布的链接（下下策）

攻击者不一定要从站内发起请求，而且就算从站内发起请求，
途径也远远不知链接一条。

  • 2.请求头Referer检测(防小白HACKER)

根据 HTTP 协议，在 HTTP 头中有一个字段叫 Referer，它记录了该 HTTP 请求的来源地址。在通常情况下，访问一个安全受限页面的请求来自于同一个网站。

验证 Referer ，就是把安全性都依赖于第三方（即浏览器）来保障，
从理论上来讲，这样并不安全。事实上，对于某些浏览器，比如 IE6 或
FF2，有一些方法可以随意篡改 Referer 值。

  • 3.请求令牌token（高级一点）
  • 4.高级请求令牌（破解难度大，影响用户体验）

图片验证码等,短信验证码,语音验证码等等

  • 5.非主流处理方式

告诉你一个不好的消息

事实上目前无法从根本解决CSRF防御问题，只能尽量构造更多障碍，提高攻击难度，构造更多障碍的同时会影响正常用户的用户体验。

浏览器同源策略

同源策略，它是由Netscape提出的一个著名的安全策略。现在所有支持JavaScript的浏览器都会使用这个策略。所谓同源是指，域名，协议，端口相同。

现有链接http://www.xxx.com/1.js，下表列出相关链接是否同源情况

站点/链接 是否同源 判断依据
https://www.xxx.com 不同源 协议不同
http://m.xxx.com 不同源 子域名不同
http://xxx.com 不同源 域名不同，此为顶级域
http://www.xxx.com:9999 不同源 端口不同
http://www.xxx.com/a.js 同源 域名，协议，端口相同

XSS注入攻击

XSS(Cross Site Scripting)，跨站脚本攻击，与SQL注入攻击类似，恶意用户，在一些公共区域（例如，论坛，评论）输入一些文本，这些文本被其它用户看到，但这些文本不仅仅是他们要输入的文本，同时还包括一些可以在客户端执行的脚本。

  • 1.反射型XSS
  • 2.持久型XSS

持久型XSS攻击一般存在于展示用户输入的页面（如评论页），这些数据会存储到网站服务器后台数据库，所以也叫存储型XSS。

链式作用域

当一个函数内部用到一个变量时，首先会在当前函数作用域中查找这个变量，如果当前函数作用域中没有，就逐层往上到上一级函数作用域中查找，一直找到window对象层，只到找到为止，如果找不到就会报变量未定义错误。这就是Javascript的链式作用域。

注意，任何函数内未加var定义的变量，函数执行的时候都会被提升为全局变量。直属于window对象，而且永驻内存不会被释放，除非手动删除，或关闭浏览器，window对象已有的该属性会被覆盖。

为用户节省流量之API接口设计

前言

移动互联网时代，在没有WIFI的情况下，用户对流量异常的敏感。往往在打开一个APP的时候，APP后台会请求服务器获取最新的数据，如果返回的数据非常多，就会很费流量。在技术层面可以利用一些技巧来设计请求的API，帮助用户节省不必要的流量消耗，也提高APP自身的可用性。这里以我们常用的QQ获取最新的好友列表为例，当然QQ具体肯定不是这么实现的，QQ客户端跟服务器的交互一般使用TCP/UDP协议建立socket长连接，这里只是记录几种使用TCP/HTTP协议短连接来实现类似功能的方案。

获取好友列表最先想到的实现方法可能就是每次开QQ时都会去拉取一次所有的好友列表数据，数据中包括用户昵称，备注，头像，个性签名，在线状态，等等。然后客户端在一直打开的情况下，还会定时轮询去拉取最新的好友列表，同步到客户端，这种可能也是最耗流量的实现方式了。下面提供两种优化的方式。

两次请求，客户端比对时间戳

还是以上面客户端获取最新好友列表为例，首先，假设客户端已经获取过一次好友列表数据，数据中包含每个用户信息的最后更新的时间戳，然后我们拉取所有好友的最后时间戳数据，跟客户端已有好友时间戳数据比对，找出不同的用户id，也许100个好友中只有10个不同，这样我们接下来只要更新下这10个用户的数据就好了，这样可以节省将近了90%的流量消耗。

这种方式有个缺点就是每次跟新数据都要请求两次服务器，一次获取用户时间戳，一次获取有差异的用户列表数据。

一次请求，服务器比对时间戳

客户端将本地好友ID和时间戳数据全部提交到服务器，服务器负责比较用户好友列表时间戳，返回有更新的用户列表。这种方式会增加服务端的压力。
